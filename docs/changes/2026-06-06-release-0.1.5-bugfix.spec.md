# Spec: Release 0.1.5 Bug-Fix Operating Rules

## Core Principles

- Treat this extension as a live product, not a refactor sandbox
- Prefer the smallest safe fix over broad cleanup
- Protect current user behavior first, especially clipboard fallback behavior
- Never close a bug based on assumption alone
- Do not mix unrelated cleanup, dependency upgrades, or UX rewrites into a bug-fix change

## Repo Risk Map

These are the highest-risk paths in this repo today:

- Markdown input selection in [src/documentInput.ts](../../src/documentInput.ts)
- Markdown rendering in [src/markdownRenderer.ts](../../src/markdownRenderer.ts)
- HTML normalization in [src/semanticHtml.ts](../../src/semanticHtml.ts)
- Rich paste formatting in [src/oneNoteHtml.ts](../../src/oneNoteHtml.ts)
- Plain-text fallback generation in [src/plainText.ts](../../src/plainText.ts)
- Clipboard write strategies in [src/clipboardWriter.ts](../../src/clipboardWriter.ts)
- Windows clipboard command generation in [src/windowsClipboardCommand.ts](../../src/windowsClipboardCommand.ts)

If a bug touches clipboard behavior, assume regression risk is high.

## Required Ticket Inputs

Every bug ticket should capture:

- Repro steps
- Observed behavior
- Expected behavior
- Environment
- User impact

Environment should include:

- Platform, currently expected to be Windows for normal maintenance work
- Windows version when relevant
- VS Code version
- Paste target, such as OneNote Desktop or Word
- Whether the repro uses selection or full document
- Sample Markdown input

If the paste target is missing, do not assume the bug is fully understood.

## Bug Classification

Classify each bug into one primary layer before changing code:

- Input layer: selection vs full document behavior
- Render layer: Markdown to HTML output
- Transform layer: semantic HTML, rich HTML styling, plain-text conversion
- Clipboard layer: Windows native, Electron, webview, or fallback behavior
- Target-editor layer: output looks correct in code, but paste result is changed by the editor

## Testing Strategy

Testing is layered. Do not jump straight to VSIX packaging for every bug.

### Layer 1: Code-Level Reproduction

Start here whenever possible.

Use the sample input from the ticket and reproduce behavior in the smallest relevant unit:

- Rendered HTML
- Normalized semantic HTML
- OneNote-targeted rich HTML
- Plain-text fallback output
- Windows CF_HTML payload when relevant

Existing test entry points:

- [test/documentInput.test.ts](../../test/documentInput.test.ts)
- [test/plainText.test.ts](../../test/plainText.test.ts)
- [test/oneNoteHtml.test.ts](../../test/oneNoteHtml.test.ts)
- [test/semanticHtml.test.ts](../../test/semanticHtml.test.ts)
- [test/htmlClipboard.test.ts](../../test/htmlClipboard.test.ts)
- [test/clipboardWriter.test.ts](../../test/clipboardWriter.test.ts)

No bug fix is complete without a regression test when the behavior can be tested in code.

### Layer 2: Extension-Host Verification

Use the current source in a VS Code Extension Development Host when:

- The bug crosses multiple modules
- You need to confirm command flow
- You need to confirm selection vs full document behavior

#### How To Verify In Extension Development Host

Use this process for normal manual verification during development:

1. Open the extension repo in VS Code
2. Start the Extension Development Host by pressing `F5`
3. If VS Code asks for a debug configuration, choose `Run Extension`
4. Wait for the new Extension Development Host window to open
5. In that new window, open a Markdown file that matches the bug repro
6. Run `Copy Markdown to Rich Text` from the editor button or Command Palette
7. Paste into the real target editor, such as OneNote Desktop or Word
8. Compare the pasted result against the expected behavior from the ticket

Use Extension Development Host as the default manual verification path before packaging a VSIX.

### Layer 3: Manual Paste Verification

Use real paste targets when the final result depends on editor behavior.

Manual paste verification is required when:

- The bug affects numbered lists, code blocks, spacing, or truncation
- The bug affects clipboard strategy behavior
- The bug looks correct in generated HTML but still fails in the editor
- The change is about to be merged or released

Preferred smoke targets:

- OneNote Desktop
- Word

### Layer 4: VSIX Packaging Check

Packaging a VSIX is a release confidence step, not the default development loop.

Use it when:

- Preparing to publish
- Checking packaged artifact integrity
- Confirming Marketplace-facing packaging behavior

## Bug-Fix Workflow

1. Reproduce the bug with the ticket sample input
2. Identify the primary layer where the bug lives
3. Add or update a regression test at the smallest useful layer
4. Confirm the test fails before the fix
5. Implement the smallest safe change
6. Run the relevant tests and then the full test suite
7. Verify in Extension Development Host when the flow crosses module boundaries
8. Perform manual paste verification when the bug depends on target-editor behavior
9. Record any target-specific limits in the ticket or changelog if needed

## Release Guardrails

Before merging or releasing a bug fix, verify these behaviors still hold:

- The command still works only for Markdown files
- Selection copy still works
- Full-document copy still works
- Plain-text fallback still works
- Windows clipboard behavior has not regressed
- No unrelated formatting path was accidentally changed

## Scope Control Rules

- One root cause per fix whenever possible
- One high-risk behavior change per commit whenever possible
- Do not reorder clipboard strategies unless the bug specifically requires it
- Do not refactor multiple formatting layers during a single bug fix
- Do not upgrade dependencies as part of a bug fix unless the dependency is the bug source
