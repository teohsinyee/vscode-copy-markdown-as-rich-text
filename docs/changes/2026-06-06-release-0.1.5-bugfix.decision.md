# Decision: Release 0.1.5 Bug Fix

## Main Decisions

- Treat `0.1.5` as a conservative patch release, not a cleanup release
- Fix high-risk bugs in separate branches first
- Use separate commits even when multiple fixes later share a branch
- Start with the highest-risk ticket first
- Validate in layers instead of jumping straight to packaged VSIX testing

## Candidate Issues

- `#1` truncation
- `#2` numbered list indentation
- `#7` ampersands in headings
- `#4` code block structure
- `#3` comparison operators and escaped characters

## Proposed Order

1. `#1` truncation
2. `#2` numbered lists
3. `#7` ampersands
4. `#4` code blocks
5. `#3` comparison operators

## Branching Strategy

- Use a separate branch for high-risk bugs
- Use a separate branch first when shared root cause is still unclear
- Allow low-risk bugs to share a branch only when the shared root cause is already clear
- Use a `release-0.1.5` integration branch only after individual fixes have been validated

## Why This Branching Model

This release contains multiple formatting and clipboard issues. The safest way to work is to keep rollback boundaries small until root causes and validation results are clear.

## Validation Decisions

- Reproduce each bug before editing code
- Add a regression test whenever the behavior can be tested in code
- Use Extension Development Host for normal end-to-end verification
- Use manual paste verification for editor-sensitive bugs
- Run relevant tests after each fix and the full test suite before integration
